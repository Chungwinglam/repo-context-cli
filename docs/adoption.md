# Adoption Guide

Use this guide to introduce Repo Context CLI into an existing repository without losing hand-authored project guidance or committing accidental secrets.

The goal is a reviewable pull request, not an automatic rewrite of repository docs.

## Who Should Use This

This guide is for maintainers who want AI coding agents to start with consistent repository context. It assumes the repository already has source code, Git history, and possibly hand-written files such as `AGENTS.md`, `PROJECT_MAP.md`, or `TESTING.md`.

Repo Context CLI is safest when adoption happens in a small branch that reviewers can inspect before any generated file becomes part of the default branch.

## Safe First Pass

Start from a clean working tree and create a short-lived branch:

```bash
git status --short
git checkout -b adopt-repo-context
```

Run a dry run first:

```bash
npx repo-context-cli pack --dry-run --for codex
```

Review the planned writes before creating files:

- `AGENTS.md` for agent-facing operating rules.
- `PROJECT_MAP.md` for indexed files, detected stacks, monorepo facts, and context size summaries.
- `TESTING.md` for detected install, build, test, lint, and format commands.
- `.repo-context/index.json` for structured context used by tools and automation.

Dry run shows planned outputs and scan warnings. It does not inspect existing target file contents to decide whether a later write will be skipped. Before generating files, check whether the repository already tracks context targets:

```bash
git ls-files -- AGENTS.md PROJECT_MAP.md TESTING.md .repo-context
```

Treat existing files and dry-run warnings as adoption decisions. Do not use `--force` during first adoption unless the team has intentionally decided to replace a hand-authored file.

## Generate Reviewable Context

After the dry run looks reasonable, generate the initial context package:

```bash
npx repo-context-cli pack --for codex --html-report --editor-config
```

This writes the default context files, an optional static HTML report, and optional editor guide files. The HTML report and editor guides are static files; they do not call an LLM, change editor settings, or modify business source code.

If you plan to enable the optional drift check immediately, generate the files with a fixed timestamp before committing them:

```bash
npx repo-context-cli pack --for codex --html-report --editor-config --generated-at 1970-01-01T00:00:00.000Z
```

If you already committed generated files without the fixed timestamp, regenerate and recommit them with the same `--generated-at` value before enabling the drift-check workflow.

Review the generated changes:

```bash
git status --short
git add --intent-to-add AGENTS.md PROJECT_MAP.md TESTING.md .repo-context
git diff -- AGENTS.md PROJECT_MAP.md TESTING.md .repo-context
```

Check for these items before opening a pull request:

- No secrets, tokens, or private source snippets are present.
- Important test and build commands are accurate.
- Unknown commands or directories are left as unknown instead of guessed.
- Existing hand-authored guidance was preserved unless the team explicitly chose to replace it.
- The generated files are useful enough for a new AI-agent session to understand the repository entry points.

## Existing Context Files

If the repository already has `AGENTS.md`, `PROJECT_MAP.md`, `TESTING.md`, `.repo-context/index.json`, `.repo-context/report.html`, or `.repo-context/editors/*`, Repo Context CLI only overwrites files it recognizes as generated by Repo Context CLI.

For hand-authored files, the default behavior is to skip the write and report a warning. Keep that default for the first pull request. If the generated version is better, copy the useful parts manually or make a separate, explicit replacement commit.

Use `--force` only when the team has reviewed the existing file and agreed that replacing it is correct.

## Pull Request Shape

Keep the adoption pull request focused:

1. Add the generated context files.
2. Add a short pull request description explaining that the files are deterministic generated context for AI coding agents.
3. Mention the exact command used, including flags.
4. Ask reviewers to check command accuracy, sensitive paths, and whether any hand-authored guidance should stay authoritative.

Avoid combining first adoption with behavior changes, dependency upgrades, or broad documentation rewrites. A small pull request makes it easier to trust the generated package.

After review, stage the files normally with `git add` before committing. `git add --intent-to-add` only makes new files visible in `git diff`; it does not stage their contents.

## Optional Drift Check

After the generated files are accepted, teams can add a check-only GitHub Action similar to this repository's `.github/workflows/context-refresh.yml`.

The recommended workflow shape is:

- Build or install Repo Context CLI in CI.
- Run `repo-context pack` with the same flags used by the repository.
- Use `--generated-at 1970-01-01T00:00:00.000Z` when tracked generated files need deterministic timestamps.
- Do not pass `--force`.
- Do not commit or push from the workflow.
- Fail only when tracked generated context files drift from the generated output.

This keeps context updates visible in pull requests instead of letting automation silently rewrite project guidance.

## Refreshing Later

Refresh the context package when the repository changes in ways that affect onboarding:

- New package manager, build system, or language stack.
- New test, lint, build, or format scripts.
- New monorepo packages or workspace layout.
- New agent operating rules.
- Release or support policy changes that should be visible to AI-agent sessions.

Use the same safe sequence each time:

```bash
npx repo-context-cli pack --dry-run --for codex
npx repo-context-cli pack --for codex --html-report --editor-config
git diff -- AGENTS.md PROJECT_MAP.md TESTING.md .repo-context
```

If the repository uses a context drift check, keep the refresh command aligned with the workflow and include `--generated-at 1970-01-01T00:00:00.000Z` before committing refreshed context files.

If a refresh would overwrite hand-authored content, stop and make an explicit maintainer decision before using `--force`.
