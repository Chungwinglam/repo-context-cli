# Example Gallery

This gallery shows practical ways to use Repo Context CLI in common AI-agent workflows. Each example starts with reviewable commands and keeps the same trust boundary: No LLM calls, no source edits, and no automatic editor setting changes.

Use these examples to decide whether the tool fits a repository before adding generated context files to the default branch.

## New project orientation

Use this when starting an AI coding session in a repository that has no agent-facing context files yet.

Recommended flow:

```bash
npx repo-context-cli pack --dry-run --for codex
npx repo-context-cli pack --for codex
```

What this proves:

- The dry run shows which context files would be created before anything is written.
- `AGENTS.md` gives the agent scoped operating rules and detected commands.
- `PROJECT_MAP.md` gives the agent a deterministic map of important files, detected stacks, monorepo facts, and rough context size.
- `TESTING.md` gives the agent the detected install, build, test, lint, and format commands when they exist.

Use this when the agent keeps asking the same first questions about project structure, package manager, or verification commands.

## Safe adoption in an existing repository

Use this when a repository may already have hand-authored `AGENTS.md`, `PROJECT_MAP.md`, `TESTING.md`, or `.repo-context/*` files.

Recommended flow:

```bash
git status --short
git checkout -b adopt-repo-context
npx repo-context-cli pack --dry-run --for codex
git ls-files -- AGENTS.md PROJECT_MAP.md TESTING.md .repo-context
npx repo-context-cli pack --for codex --html-report --editor-config
git diff -- AGENTS.md PROJECT_MAP.md TESTING.md .repo-context
```

What this proves:

- The first pass is reviewable before generated files are committed.
- Existing hand-authored files are skipped unless they were previously generated by Repo Context CLI or `--force` is intentionally used.
- The optional HTML report gives reviewers a browser-readable summary without JavaScript.
- The optional editor guides are static Markdown files and do not modify Cursor, VS Code, or editor-owned settings.

Use this for first adoption pull requests where maintainers need to inspect command accuracy, sensitive paths, and whether existing guidance should stay authoritative.

## Context refresh pull request

Use this when a repository already tracks generated context files and the project structure has changed.

Recommended flow:

```bash
npx repo-context-cli pack --dry-run --for codex
npx repo-context-cli pack --for codex --html-report --editor-config --generated-at 1970-01-01T00:00:00.000Z
git diff -- AGENTS.md PROJECT_MAP.md TESTING.md .repo-context
```

What this proves:

- The refresh is explicit and visible in a pull request.
- The fixed `--generated-at` value keeps tracked generated files stable across repeated CI checks.
- Reviewers can inspect only the changed context files instead of re-reading the whole repository.
- A check-only GitHub Action can fail on drift without committing, pushing, or running `--force`.

Use this after adding packages, changing scripts, introducing a new stack, or updating agent operating rules.

## MCP and editor-assisted workflows

Use this when an MCP-compatible client should consume repository context without writing files during the session, or when a team wants explicit static editor guides for local setup.

No-write MCP flow:

```bash
npx repo-context-cli mcp
```

Editor guide generation flow:

```bash
npx repo-context-cli pack --editor-config
```

What this proves:

- `repo-context mcp` exposes a read-only stdio MCP server with `get_repo_context`.
- MCP responses use the same redacted, dry-run context generation path as file output.
- For a no-write session, start only the MCP server.
- Static editor guides explain how Cursor, VS Code, and generic AI-editor workflows can consume the generated context package.
- When you want editor guides, generate them explicitly with `pack --editor-config`.
- The workflow keeps editor setup explicit; Repo Context CLI does not create `.vscode/settings.json`, `.cursor/rules`, or other editor-owned files.

Use this when teams want the same repository facts available to AI-agent sessions without silently changing local editor configuration.

## Choosing an example

- Start with New project orientation for a fresh repository or a quick evaluation.
- Start with Safe adoption in an existing repository when generated files need maintainer review.
- Use Context refresh pull request after repository structure or commands change.
- Use MCP and editor-assisted workflows when repository facts should be available through tools without immediate file generation.
